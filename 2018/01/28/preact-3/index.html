render.js

Preact.render 方法将 component 树挂载到 DOM 节点, 类似 ReactDOM.render。

前两个参数和 ReactDOM 的 render 方法类似，不同的是这里提供了第三个参数 merge，如果 merge 存在并且是 parent 的子节点，那么会通过 diff 算法来更新 merge。

diff.js

diff 算法是整个框架的核心，决定了 Preact 如何判断是否要更新或重建一个节点(节点的属性)。通俗的讲，就是如何以最小的代价通过 vnode tree 来更新现有的 dom tree。

diff 的参数如下:

第一个参数 dom 是本次 diff 需要更新的 dom tree。可以为 null。为 null 时即为创建dom。
第二个参数是通过 createElement 返回的虚拟 dom 即 vnode，代表了本次 diff 返回的 dom 的结构。
第三个参数 context 和 react 中的 context 类似，是祖先节点通过 getChildContext 方法传递下来的prop，默认值为一个空对象
第四个参数 mountAll 代表是否要 mount 整个树，这关系到 component 实例在 diff 过程中是否强制调用 componentWillMount 和 componentDidMount。
第五个参数就是整个 tree 挂载的 container
第六个参数指定本次 diff 的 vnode 是不是某 component 实例的root节点。如果是 root 则需要在 root diff 产生的 dom 被替换到文档流之后才能调用 flushMounts 方法，否则 componentDidMount 可能会取不到 dom 节点。

diff 方法首先进行两个判断:
1.是否是在 svg 元素内。这个关系到后面是创建 svg 元素还是 dom 元素
2.是否是 hydrating。 server render 的 dom 或者是其他方法(例如直接append)插入的 dom 需要 hydrate。

之后进入内部的 idiff(dom, vnode, context, mountAll, componentRoot) 方法。这个方法内部是真正的 diff 逻辑。该方法的 diff 逻辑总结如下:

  1. 如果 vnode 为 null 或者 vnode 是布尔值，则设 vnode='', 即空TextNode。(因为 typeof '' === 'string', 所以进入下面的2)。
  2. 如果 vnode 是 number 或者 string 类型，而对应的 dom 是Textnode 的话，更新 dom.nodeValue = vnode, 如果不是 Textnode 则创建 textNode 并替换掉原 dom 节点。返回 dom 并结束本轮 idiff。
  3. 如果 vnode 是一个 component，则通过 buildComponentFromVNode 方法进行 component diff。
  4. 如果 dom 不存在或者 dom 的 nodeName 和 vnode.nodeName 不匹配，则根据 isSvgMode 来创建新 element。如果是后一种情况，则把原先 dom 的 children 放入新建 element 内, 以便之后进行 innerDiffNode。如果原先 dom 有父节点，则用新建的 element 替换上去。 可见这一步的代价是非常大的。会进行很多 DOM 节点操作。

由于2和4两种情况都存在旧的 dom 节点被替代的现象，所以都会通过 recollectNodeTree(node, unmountOnly) 方法来回收节点。

经过上面4步之后如果 vnode 没有 children，则通过 diffAttributes(out, vnode.attributes, props) 方法更新 diff 后的 dom atrtibute/property/ref，同时更新 dom[ATTR_KEY], 然后返回 dom 节点，一次 diff 就结束了。

如果 vnode 存在 children 属性，并且 dom 也有 children，则进入 innerDiffNode 方法：

其中参数 dom 表示上一步包含 children 的 dom 元素, vchildren 是上一步 vnode 的 children。该方法主要是对 dom 的 children 和 vnode 的 children 做 diff。这个方法逻辑总结如下:

1.首先根据 dom children 是否有 key 属性（child._component.__key 或者 child[ATTR_KEY].key）将他们分成一个有key的 key-value 对象和一个无key的 children 数组。

2.然后对 vchildren 做循环。如果 vchild 有 key 属性，则从第1步产生的 key-value 对象中找到对应key的 dom child。如果 vchild 没有 key，则在无key的 children 数组中找到 第一个相同type 的 child。

值得注意的是 相同type 的判断是通过isSameNodeType(node, vnode, hydrating) 进行的。isSameNodeType 的逻辑是：

  • 如果 vnode 是 string 或 number 类型，node 是 textNode 类型，则认为是相同类型。
  • 如果 vnode.nodeName 是字符串，则通过 nodeName 判断是否是相同类型。
  • 如果 vnode 是一个 component 则比较两者是不是同个构造函数(node._componentConstructor===vnode.nodeName) 如果是则认为是相同类型。
  • 如果是 hydrating 或者父节点有 dangerouslySetInnerHTML 这个 attribute 则认为是相同类型。
  • 其他情况都不是相同类型。

从上面的逻辑可以看出，给同类型的 children 加 key 是非常重要的。通过 key 可以准确的找到需要做 diff 的 dom child， 避免了 isSameNodeType 的判断。由于无 key 时使用第一个匹配的 child，所以如果某次 update 只更新 children 顺序的话，不加 key 就会产生很多不必要的 diff 更新，继而产生不必要的 dom 操作。

通过上面的2步判断，已经获得了对应的 child 和 vchild，那么就会再次进入 idiff 方法： idiff(child, vchild, context, mountAll) 对 child dom 做 diff，直到没有 vchild 为止。

idiff 方法最后再次通过 recollectNodeTree(keyed[i], false) 方法回收了所有本次diff未被匹配到的 child (并且从文档流中 remove 掉)。
