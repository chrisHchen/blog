preact 源码(总结篇)

diff 做了什么？

用作者的原话就是:

Apply differences in a given vnode (and it’s deep children) to a real DOM Node.

其中的 differences 本质上其实是 component 实例的 base 属性和 component 实例的 render 方法执行后返回的 vnode 之间的 difference。 其中base 属性就是上一次 diff 后生成的 Dom 节点。 所以就是用上次的 dom 和本次的 vnode 做 difference/diff。两种状态作比较，通过 diff 方法内的一整套规则，用最小的代价来做更新。

diff 的流程

初次render:

preact.render() =>diff() => idiff() =>[ buildComponentFromVNode => setComponentProps <=> renderComponent <=> component.render() => diff() => ...] => innerDiffNode() => idiff() => … until no vnode left, return dom root.

如果是 update, 那么直接从 renderComponent 开始执行。

其中蓝色部分是 component diff 的过程，会触发我们熟悉的的一系列生命周期函数。他们执行的先后顺序如下:

componentWillMount: parent => child
componentDidMount: child => parent
componentWillUpdate: parent => child
componentDidUpdate: child => parent
componentWillUnmount: parent => child

componentWillReceiveProps: parent => child
shouldComponentUpdate: parent => child

值得注意的是，componentWillReceiveProps 是在 setComponentProps()中被执行, 而且是在 renderComponent/enqueueRender 之前执行的, 所以在 componentWillReceiveProps 中可以调用 setState 来改变组件状态，新状态会在 renderComponent/enqueueRender 中被体现。

而 componentWillUpdate 是在 renderComponent 中被执行，如果在 componentWillUpdate 中调用了 setState，会导致正在 render 的实例被再次推入 render 队列，出现死循环。同理 render 时也不能调用 setState 。

diff 过程中的几个容易忽视的点

  • 如果出现新建 dom 节点的情况，且老的节点有父节点，那么一定要用新建节点替换父节点下的老节点。同样的，如果老节点有子节点，也要将子节点 append 到新建节点。

  • 当有 hoc 的情况时，所有祖先 component 的 base 属性都指向最后那个child产生的 dom 节点，而这个 dom 节点的 _component 属性指向最高的祖先 component 实例。(_componentConstructor 属性也同样指向的是最高祖先的构造函数)

collectComponent方法: 回收实例

collectComponent 函数会将 unmount 的 component 实例按照构造器来分类回收。当新建 component 实例时, 会在回收器中拿出一个同构造函数的实例，将实例对应的 dom (nextBase属性)赋值给新实例的 nextBase 属性。这样可以重用 dom 节点，减少 createElement 操作。

当然这样做的前提是认为同一个构造器大多数情况下总是会 render 出相同 type 的 dom。
