react-redux 源码分析 [v5.0.7]

进入源码之前先看一下 npm script 里用于打包的几个命令：
其中 umd 模块是用的 rollup 打包 ： build:umd:min 和 build:umd
而 es 模块 和 commonjs 模块都是用的 babel直接打包: build:es 和 build:commonjs。

其中的 BABEL_ENV 环境变量会对应到 .babelrc 的 env 属性，对每个不同的 BABEL_ENV 会配置不同的 babel plugins。
当 BABEL_ENV = rollup 时，由于 rollup 用到了 rollup-plugin-babel 所以最终还是会定位到 .babelrc, 并且此时 env=rollup。

打包的入口为 src/index.js, 它导出了一个对象：

Provider

先看 Provider => src/components/Provider.js。

Provider 利用 react 的 context api 是得传入的 store 对象能够被后代 children 获取到。

context api 要实现 getChildContext 这个实例方法以及定义 Provider.childContextTypes 这个静态对象：

可见传入的是 context 对象上有两个key： storeKey 和 subscriptionKey。 storeKey 默认值为 'store', subscriptionKey 默认值为 'storeSubscription'

最后导出 Provider。

createProvider

createProvider 其实是提供了一个自定义 context 对象上 storeKey 和 subscriptionKey 的方法。只有在存在多个store时才会用到自定义 key，所以createProvider 函数是基本不会用到的。

connect、connectAdvanced

connect 函数接受 mapStateToProps, mapDispatchToProps, mergeProps, 以及一个 option对象。

最终通过各个 factory 函数将他们包装为可以被 selectorFactory 接受的函数。这里的每个 factory 函数主要做了一件事，就是判断 mapStateToProps 和 mapDispatchToProps 是否依赖第二个参数 ownProps。

然后将参数传入 connectAdvanced。 connectAdvanced 接受 selectorFactory 和 options 最终返回一个 HOC。HOC 将接受 WrappedComponent 并返回最终包装后的 Component:

Connect 通过 context 接受到传递下来的 store 对象。并且通过 selectorFactory 生成 this.selector 对象。

this.selector 对象负责 store 更新后生成新的 props, 并传递给 WrappedComponent。 同时 selectorFactory 用闭包缓存了上一次的 store, stateProps, dispatchProps, ownProps, 以及最终的mergedProps, 在 option.pure 的情况下，会通过 option 里的几个比较函数: areStatesEqual, areOwnPropsEqual, areStatePropsEqual, 来判断新旧状态是否有差别，进而判断是返回新对象还是旧对象。进而影响 this.selector.shouldComponentUpdate 的值。这样，当每次 store 更新时，并不是所有 connect 的 component 都会 re-render，只有 mapStateToProps, mapDispatchToProps, mergeProps，这几个函数的结果结合 ownProps 后通过 selectorFactory 的比较逻辑，确实发生变化的才会 re-render。

另外，re-render 的逻辑如下:

  1. 在 componentDidMount 执行 this.subscription.trySubscribe(), 通过this.store.subscribe(this.onStateChange) 注册 onStateChange 函数。并执行 forceUpdate。
  2. 每次 store 变化后会自动执行 onStateChange函数。
  3. onStateChange 内部执行 this.selector.run(this.props) 并通过this.selector.shouldComponentUpdate判断最终的 props 是不是变化了。
  4. 如果有变化，则执行 this.setState(dummyState) 来进行 re-render。 其中 const dummyState = {}。 类似于forceUpdate
