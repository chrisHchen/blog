draft-plugin 源码

draft-js 是 facebook 推出的基于 React 和 immutable 数据模型的富文本编辑器构建框架。(文档)

draft 本身是通过改变 Editor 组件的 editorState 这个 prop 来渲染出每次内容改变的结果。我们可以在 onChange 事件内生成新的 editorState 对象(editorState 是 immutable 的，智能生成新的，不能修改)，来改变 Editor 的 editorState。也可以通过给 Editor 组件传入各种 prop 来改变他的默认行为，生成符合我们要求的 editorState。

而 draft-js-plugins 则提供了一种插件机制，将每种独立行为都做成一个插件，用配置插件的方式来灵活的生成符合需求的富文本编辑器。

这里主要看下 draft-js-plugins 如何将插件化的配置植入到 draftjs内部的，所以不会涉及具体插件的实现。

draft-js-plugins-editor/src/index.js

这个文件导出了最终接受 plugins 的 PluginEditor 组件。

这里的 Editor 就是 Draft 的 Editor 组件。

首先在构造函数中会依次调用每个 plugin 的 initialize方法，传入 getPluginMethods方法返回的对象以提供 plugin 初始化需要的一些参数。

然后在 componentWillMount 中通过 resolveDecorators 函数剥离出 plugins 中的 decorators，通过 EditorState.set 静态方法返回带 decorator 的新 plugin，最后通过 onChange 方法设置新的 editorState。

onChange 方法会依次调用每个 plugin 的 onChange 方法，上个 plugin 的 onChange 方法返回的 editorState 作为下一个 plugin 的 onChange 入参。可以看出 plugin 如果有 onChange函数，则必须返回一个新的 editorState 类型的对象。

resolveDecorators 方法首先将直接作为 decorators prop 传入的 decorator 数组和作为 plugins prop 中的 decorators 属性间接传入的 decorator 数组合并(并做了数组扁平化处理)。 然后通过 decoratorIsCustom 过滤出不是非custom的decorator，因为非 custom 的 decorator 需要通过 CompositeDecorator 构造函数来生成 decorator 实例。 这个在 draftjs 的decorators文档里有说明。然后将生成的 compositeDecorator 实例连同 custom decorator 一起，生成 MultiDecorator 的实例，作为最终的 decorator 实例返回。

MultiDecorator 是实现了 DraftDecoratorType Flow type definition 的类。

componentWillReceiveProps 周期函数中的操作不太清楚原因…

componentWillUnmount 函数中会一次执行每个 plugin 的 willUnmount 函数。

createPluginHooks 函数会将 plugin 的属性分为三种：

  • 以 on 开头的事件型属性 (onChange除外)
  • 以 handle 开头的 handle 属性
  • 以 Fn 结尾的 Fn 属性

对于事件型属性，通过 createEventHooks 函数返回处理句柄，句柄会依次执行 plugin 的对应事件句柄，直到某个句柄返回 true 为止。可以看出如果某个 plugin 想终止某事件被后续plugin的句柄执行，只需要让自身定义的句柄返回 true 即可。

对于 handle 类型的属性，通过 createHandleHooks 返回最终处理函数，处理函数会依次执行 plugin 对应的 handle 函数，直到某个 handle 返回 ‘handled’ 字符串。name整个处理函数将返回 ‘handled’， 否则返回 ‘not-handled’。这里的应用类似于 draft 的 handleKeyCommand 这个 prop。(文档)

对于 Fn 类型的属性，通过 createFnHooks 函数处理后返回最终的处理函数。这里对 blockRendererFn 和 blockStyleFn 两种情况做了特殊的 merge 处理。其他的 fn 则会依次执行 plugin 对应的 fn ，直到某个 fn 返回值不为 undefined 为止，处理函数返回这个 fn 的返回值，否则返回 false。

另外，对 customStyleMap 和 blockRenderMap 这两个prop 也做了依次 merge 每个 plugin 对应属性的操作。

可见 PluginEditor 组件其实是对每个 plugin 里相同的属性分类，然后对这些属性做 流式处理。让每个 plugin 都依次处理前一个 plugin 返回的 editorState，最终返回处理完成后的新 editorState。
